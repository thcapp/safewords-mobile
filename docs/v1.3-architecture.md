# Safewords v1.3 Architecture

Status: **shipped** (Android v1.3.1 / versionCode 15 on Play internal, iOS v1.3.1 / build 7 on TestFlight)
Last updated: 2026-05-01

This doc explains the v1.3 architecture: configurable verification primitives, Plain-as-default UX, demo mode, and native printable cards. Read after `docs/feature-spec.md` and the per-platform architecture docs.

## The shift in mental model

Before v1.3, "Safewords" meant *one rotating word per group, shared by all members*. That's still the default, but v1.3 makes the verification primitive a first-class group concept. A group can enable one or more primitives; the call-screen UX adapts without exposing knobs.

The single sentence that drives every decision:

> **Configure the primitive in the shared group model, get the same experience on the front.**

Both apps read the same `Primitives` config, derive everything deterministically from the seed, and render the same UX. There is no server. There is no runtime backend.

## Primitives

Five primitives are defined; v1.3 ships four (one is deferred):

| Primitive | Shape | Source | Status |
|---|---|---|---|
| **Rotating word** | TOTP-derived adjective+noun+number, all members see the same word at the same moment | `HMAC-SHA256(seed, counter)` | shipped |
| **Numeric** | TOTP-derived 6-digit code, same time-bucket as rotating word | `HMAC-SHA256(seed, counter)` ظْ dynamic-truncate ظْ `% 1_000_000` | shipped |
| **Static override** | One frozen word per group, derived from seed, never rotates | `HMAC-SHA256(seed, "safewords/static-override/v1")` ظْ standard word derivation | shipped |
| **Challenge / answer** | 100-row deterministic group table; ask one phrase, expect a paired phrase back | per row: `HMAC-SHA256(seed, "safewords/challenge-answer/v1/{ask\|expect}/{rowIndex}")` ظْ standard word derivation | shipped |
| **Pair (per-contact rotating)** | Asymmetric per-counterparty rotating pairs | future: seed + time + counterparty | **deferred to v1.4+** |

### Derivation specs

All four shipped primitives use HMAC-SHA256 with a 32-byte seed. All map either through the standard word derivation (offset-based extraction from the hash) or through dynamic truncation for numeric output.

**Rotating word** (unchanged from v1.0):
1. `counter = floor(timestamp / intervalSeconds)`
2. `hash = HMAC-SHA256(seed, big-endian-int64(counter))`
3. `offset = hash[31] & 0x0F`
4. `adj = ((hash[offset] & 0x7F) << 8 | hash[offset+1]) % 197`
5. `noun = ((hash[offset+2] & 0x7F) << 8 | hash[offset+3]) % 300`
6. `number = ((hash[offset+4] & 0x7F) << 8 | hash[offset+5]) % 100`
7. Render `"{adj} {noun} {number}"`

**Numeric** (RFC 4226 dynamic truncation pattern):
1. Same `counter`, same `hash` as rotating word
2. `offset = hash[31] & 0x0F`
3. `code = (((hash[offset] & 0x7F) << 24) | ((hash[offset+1] & 0xFF) << 16) | ((hash[offset+2] & 0xFF) << 8) | (hash[offset+3] & 0xFF)) % 1_000_000`
4. Render exactly six zero-padded decimal digits

**Static override**:
1. `hash = HMAC-SHA256(seed, "safewords/static-override/v1".bytes)`
2. Apply the standard word derivation (same as rotating word, steps 3ظô7)

**Challenge / answer**, per row index `r` and table version `v`:
1. `askHash = HMAC-SHA256(seed, "safewords/challenge-answer/v{v}/ask/{r}".bytes)`
2. `expectHash = HMAC-SHA256(seed, "safewords/challenge-answer/v{v}/expect/{r}".bytes)`
3. Apply standard word derivation to each, yielding `(ask, expect)`

The cross-platform contract is `/shared/primitive-vectors.json` ظ¤ every derivation has fixture vectors against `seedA = 01..1f20` and `seedB = ff..ff`.

## Group config schema

`/shared/migration-vectors.json` defines the on-disk shape:

```json
{
  "schemaVersion": 2,
  "id": "uuid",
  "name": "Family",
  "seed": "<hex>",
  "primitives": {
    "rotatingWord": {
      "enabled": true,
      "intervalSeconds": 86400,
      "wordFormat": "adjective_noun_number"
    },
    "challengeAnswer": { "enabled": false, "tableVersion": 1, "rowCount": 100 },
    "staticOverride": { "enabled": false, "derivationVersion": 1, "printedAt": null }
  },
  "members": [...]
}
```

**Storage rule**: store config only, never derived secrets. The override word, challenge/answer table, and recovery phrase are recomputed on demand from the seed.

**Migration rule**: a v1.2 group on disk has no `schemaVersion` and no `primitives` field. The reader expands such a group to `rotatingWord.enabled = true` with the legacy `intervalSeconds`, all other primitives disabled. Pass-through if the group is already v1.3.

## UX architecture

### Plain Mode is the default

Pre-v1.3, the tabbed UI was the front door and Plain Mode was an opt-in accessibility variant. v1.3 inverts that: Plain Mode is the default home for every user. The big-word, big-button layout is the front door. The tabbed UI demotes to "Advanced view" ظ¤ opt-in via Settings.

Reasoning: under stress, the home screen should already be the call-decision screen. There's no second tab to hunt for.

### View toggle

A gear icon (or equivalent) in the Plain home top-right opens Settings, which lives under the Advanced view. The toggle is sticky ظ¤ once a user picks Advanced, they stay there until they toggle back.

Storage: `advanced_view_enabled` boolean in EncryptedSharedPreferences (Android) / UserDefaults (iOS app group).

### Verify tab is conditional

The Verify tab only earns its slot when the two parties have *different* screens. For rotating-word-only groups, Home IS the verification screen ظ¤ they say a word, you compare to what's on your screen, done.

For challenge/answer or static-override groups, the Verify tab becomes a Match-confirm sheet:
- "Ask: APPLE / Expect: ORANGE" + binary `They said it` / `Doesn't match` buttons
- Random row per session; "use a different row" reroll
- Result panel directs user to hang up + call back on a known number on mismatch

Empty-state copy when no primitive needs Verify: *"Verify isn't needed for {group} right now ظ¤ both phones show the same word. Add challenge/answer or a static override in Settings ظْ Verification if you'd like a tap-to-confirm flow."*

## Native printable cards

Cards render natively on each platform. No webview, no network round-trip. The `/shared/safety-card-copy.json` file holds copy strings keyed by template name; each platform renders from that copy plus the group's runtime data.

### Card types and sensitivity

| Card | Sensitivity | Gate | What it shows |
|---|---|---|---|
| Protocol | Low | None | "Ask: what's our word? / Don't say it first / Hang up if wrong" + install QR |
| Static override | High | Biometric | The override word + warning copy |
| Challenge/answer wallet (24 rows) | High | Biometric | First 24 ask/expect pairs, fits a fold-card |
| Challenge/answer protocol (100 rows) | High | Biometric | Full table, fits a folded letter |
| Recovery phrase | High | Biometric | 24-word BIP39 phrase + warning |
| Group invite | High | Biometric | The QR; treated as seed-equivalent until expiring invites exist |

### Render pipeline

**Android**: Compose-laid-out content ظْ `Bitmap` via Canvas ظْ `androidx.print.PrintHelper.printBitmap()` to system print dialog. Source: `app/src/main/kotlin/com/thc/safewords/print/CardRenderer.kt`.

**iOS**: SwiftUI view ظْ `UIGraphicsImageRenderer` ظْ image ظْ `UIPrintInteractionController` (or `UIActivityViewController` for AirDrop / Save PDF). Source: `Safewords/Print/CardRenderer.swift`.

Both platforms produce letter-size 300dpi bitmaps for high-sensitivity cards (8.5" ├ù 11") so the printed result is sharp. Wallet cards use 3.5" ├ù 2.0" landscape orientation.

### Biometric gate

High-sensitivity cards prompt for biometric/credential auth before rendering. Android uses `BiometricService` (wraps `BiometricPrompt` with `DEVICE_CREDENTIAL` fallback). iOS uses `LocalAuthentication.deviceOwnerAuthentication`. If no biometric is enrolled, the gate falls through (Android) or surfaces an error (iOS) ظ¤ printing without auth is preferable to blocking the legitimate user from their own card.

## Demo mode

Demo mode lets a user explore the app before committing to a real group. v1.3.1 addition.

### Mechanism
- "Try without a group" option on onboarding panel 1
- Routes to `enterDemoMode()` which inserts a hardcoded demo group:
  - **ID**: `00000000-0000-0000-0000-00000000d013`
  - **Name**: `Demo ┬╖ TIGER`
  - **Seed**: 32 bytes spelling `TIGER-DEMO-SAFEWORDS-V13-DEMO-!!` in ASCII (intentionally readable so anyone reverse-engineering sees this is not a real seed)
  - **Primitives**: rotating word + challenge/answer + static override all enabled, so users see every v1.3 surface working
- Plain home shows a "DEMO MODE / Set up your real group ظْ" banner; tapping exits demo + switches to Advanced view, lands on real onboarding
- `createGroup` and `joinGroup` auto-call `exitDemoMode()` first

### Cross-platform parity
Both apps use the same demo seed bytes and the same demo group ID. If two devices are in demo mode at the same time on the same date, they show the same word.

### Storage
- Demo seed is **never** written to Keychain (iOS) or `SecureStorageService` (Android)
- `demo_mode` flag persists in EncryptedSharedPreferences / UserDefaults App Group so the demo state survives launches
- `resetAllData()` clears the flag

## Defensive recovery loading

v1.3.1 also hardened the BIP39 wordlist load path. Pre-1.3.1, a missing `bip39-english.txt` resource would `fatalError` and crash the app on the recovery-phrase generation step. Post-1.3.1:
- `Bip39.init` throws a typed error if wordlist is missing or has wrong count
- `RecoveryPhrase` (singleton) wraps initialization in `Result<>`
- Callers fall back to hex seed display when the wordlist is unavailable

## What's NOT in v1.3

- **Pair mode** (per-contact rotating C/A) ظ¤ deferred until member identity is crypto-stable across devices and recovery
- **Agent Circle** ظ¤ separate repo `safewords-agent`, not part of consumer app
- **Custom wordlists per primitive** ظ¤ v1.4+ might add `wordFormat` extensions; for now all primitives reuse the same adj/noun/number space
- **Expiring invites** ظ¤ until they exist, the group invite QR is treated as seed-equivalent (high sensitivity)
- **App Store / Play Store listing copy update for primitives** ظ¤ post-v1.3 marketing decision
