# Safewords v1.3 — Best-in-Class Design Brief

Status: **Draft v4 (codex lock cleanup)** — pending final ack
Date: 2026-04-30

## Frame

The bar for v1.3 is *best-in-class*, not "pick one track." Codex's
expansion proposal raised five strong directions; the user was clear
that we should not narrow to one. Instead the platform learns to host
all of them behind a uniform UX.

The single sentence that drives every decision below:

> **Configure the primitive in the shared group model, get the same
> experience on the front.**

Codex redline: avoid "backend" language. Safewords' core claim is
offline-first/no-server. The source of truth is the local group model
plus deterministic derivations from the group seed. If a future hosted
admin surface exists, it writes this same portable group config; it is
not a runtime backend dependency.

## What changes in mental model

Pre-1.3, "Safewords" meant "rotating TOTP word, one per group." That
shape is the default but no longer the only shape. v1.3 introduces a
group's *verification primitives* as a first-class concept. A group can
have one or more primitives active, and the call-screen UX adapts
without showing knobs.

### Primitives

| Primitive | Shape | Source | When best |
|---|---|---|---|
| **Rotating word** | One TOTP-derived word everyone in the group sees at the same moment | seed + time | family-wide trust, household call check |
| **Challenge / answer** | Asymmetric prompt/response; I ask a prompt, they answer the paired response | deterministic table from seed + table version | per-relationship verification, printed tables, assisted workflows |
| **Static override** | One frozen word that never rotates, printed on a card | deterministic from seed | "I'm in trouble" cases, fridge cards, caregivers |
| **Numeric** | TOTP rendered as digits instead of words | seed + time | preference / accessibility |
| **Pair (per-contact rotating)** *(deferred to v1.5+)* | Asymmetric pair table that rotates | seed + time + counterparty | high-volume professional verification |

A group enables one or more of these. Active primitives drive (a) what
the call screen shows, (b) what contextual actions appear, (c) what cards
are printable.

Codex redline: do not make stressed users select a counterparty before
they can verify a call. v1.3 challenge/answer should be group-table
based first. Counterparty-keyed derivation belongs later when membership
identity is stable across devices and recovery.

### Plain Mode → default

The current "Plain Mode" stops being an accessibility branch. It
becomes the default home experience for every user. The big-word,
big-button layout is the front door. The current tabbed/standard UI
becomes "Advanced view" — opt-in via Settings, mainly for power users
inspecting multiple groups.

Reasoning: if you're under stress, the home screen should already be
the call-decision screen. There's no second tab to hunt for. Plain Mode
was always the right design; we just had it labeled as a fallback.

### Verify tab → conditional

The Verify tab only earns its slot when the two parties have *different
screens*. For rotating-word-only groups, Home IS the verification
screen — they say a word, you compare to what's on your screen, done.
Typing isn't useful; nothing to verify against.

For challenge/answer or static-override groups, the Verify tab becomes
"they answered with X — does it match my expected response?" — a
buttoned/tap UX, not a free-text input.

Tab visibility is a function of the active group's primitives.

## UI: home (the new front door)

```
┌──────────────────────────┐
│  Family · ⏱ 14m left    │  ← group name + countdown for rotating
│                          │
│       T I G E R          │  ← word, huge type
│                          │
│   Ask: "What's our       │  ← guidance, always present
│        word?"            │
│                          │
│  ╔══════════════════╗    │  ← challenge/answer ONLY if enabled
│  ║ Verify response  ║    │     for this group
│  ╚══════════════════╝    │
│                          │
│   [ Switch group ]       │  ← bottom nav minimized
│                          │
└──────────────────────────┘
```

For challenge/answer groups, the home tile has a contextual action:
"Challenge someone". The app picks or lets the user pick a row, then
shows:

- **Ask:** APPLE
- **Expect:** ORANGE

The other party can answer from their app or from a printed challenge
table. This is intentionally not counterparty-specific in v1.3.

For static-override-enabled groups, an optional second tile shows the
override is available, no value displayed unless the user explicitly
reveals.

## Group config

```jsonc
{
  "id": "uuid",
  "name": "Family",
  "schemaVersion": 2,
  "seed": "<encrypted in keystore>",
  "primitives": {
    "rotatingWord": {
      "enabled": true,
      "intervalSeconds": 86400,
      "wordFormat": "adjective_noun_number"  // or "numeric"
    },
    "challengeAnswer": {
      "enabled": false,
      "tableVersion": 1,
      "rowCount": 100                         // derived, not stored
    },
    "staticOverride": {
      "enabled": false,
      "derivationVersion": 1,                 // derived, not stored
      "printedAt": null
    }
  },
  "members": [...]
}
```

Default for new groups (proposed, needs user confirm): rotating word
only. Static override and challenge/answer are opt-in via Settings →
Group → Add primitive.

Codex redline: do not store derived secrets (`staticOverride.word`,
challenge rows, recovery phrase) in group metadata. Store only config
needed to reproduce them from the seed. If a value must be revocable
without rotating the seed, it is no longer seed-only recovery and needs a
separate backup story.

## Printable cards — native, no network

Web app already does printables for desktop. Mobile renders its own,
natively, offline. No deep-link to safewords.io for printing — that
would violate the zero-network claim that's the strongest privacy lever
we have.

Native rendering primitives:

- **iOS**: SwiftUI layout → `UIGraphicsImageRenderer` → image →
  `UIPrintInteractionController` (or share sheet for AirDrop / Save
  PDF). ~50 LoC.
- **Android**: Compose → bitmap → `PrintHelper.printBitmap` (built-in
  AndroidX). ~40 LoC.

### Card types

Each primitive can have a card. Sensitivity tier dictates the gate.

| Card | Sensitivity | Gate | Renders |
|---|---|---|---|
| Protocol card (instructional) | Low | None | "Ask: what's our word? / Don't say it first / Hang up if wrong" + install QR |
| Static override card | High | Biometric | The override word itself, with explicit "anyone with this card can act as you" warning |
| Challenge/answer table | High | Biometric | The full pair table |
| Recovery phrase card | High | Biometric | 24-word BIP39, "anyone with this restores the group" warning |
| Group invite card | High | Biometric | The QR for joining; seed-equivalent unless/until expiring invites exist |

Shared copy lives in `/shared/safety-card-copy.json`. Each platform
renders natively from those copy strings + the group's runtime data.

Codex redline: "expires per group config" is not true for current invite
QRs. A join QR that contains the group seed is high sensitivity. Either
gate it like recovery material or defer invite cards until signed/expiring
invite payloads exist.

## What ships in v1.3

### Must

- [ ] Plain Mode promoted to default; current tabbed UI → "Advanced"
  setting (Android + iOS)
- [ ] Group config gains `primitives` object; v1.2 groups migrate with
  `rotatingWord.enabled = true`
- [ ] Verify tab made conditional on active primitives
- [ ] Static override primitive: enable/disable, derive, persist config,
  reveal/reprint
- [ ] Native printable cards: protocol card + static override card
- [ ] Shared copy file `/shared/safety-card-copy.json`
- [ ] `/shared/primitive-vectors.json` for cross-platform tests of
  primitive derivations
- [ ] `/shared/migration-vectors.json` for v1.2 → v1.3 schema defaults
- [ ] Recovery phrase printable card, because v1.2 already made BIP39 the
  shared recovery contract and it belongs in the sensitive-card family

### Should

- [ ] Challenge/answer primitive: deterministic shared table, settings
  UI, "Challenge someone" sheet
- [ ] Group invite card only if treated as high sensitivity, or if
  expiring invite payloads are added first

### Should (claude v3 add)

- [ ] **Numeric word format** — pull forward from v1.4+. It's
  `HMAC(seed, time-bucket)` rendered as a 6-digit code from the same
  bytes we already extract for word indexing; ~half a day per platform.
  Hits the Supercode "Stop AI Scams" search-term territory and gives
  users who prefer numbers an offline-equivalent to TOTP authenticators.
  Group config: `rotatingWord.wordFormat = "numeric"` selects it.

### Defer to v1.4+

- [ ] Pair mode (per-contact rotating/counterparty-keyed)
- [ ] Agent Circle (separate repo `safewords-agent`, not consumer app)

## Cross-platform contract

Same as today: anything that derives from the seed must produce
byte-identical results on iOS and Android. New test vectors required:

- `/shared/primitive-vectors.json` — static override derivation,
  challenge/answer table derivation, numeric derivation, and card copy IDs.
- `/shared/migration-vectors.json` — v1.2 group input to v1.3 group output,
  including default `primitives` expansion.

For challenge/answer v1.3, derivation must be deterministic from
(seed, tableVersion, rowIndex). Counterparty-keyed derivation is deferred
until member identity is stable across devices and recovery.

For static override, derivation is deterministic from
`HMAC-SHA256(seed, "safewords/static-override/v1")`, mapped through the
**same adjective/noun/number phrase space** as rotating words. Decision
(claude v3): no dedicated wordlist. Users learn one shape; cards look
consistent; one less wordlist to maintain. If we later want shorter
override forms (e.g., 4 letters for caregiver cards), that becomes a
v1.4+ `wordFormat` extension, not a parallel wordlist.

For numeric: use the same counter and HMAC as rotating words, then
standard dynamic-truncation-style extraction:

1. `counter = floor(timestamp / intervalSeconds)`
2. `counterBytes = int64 big-endian`
3. `hash = HMAC-SHA256(key = seed, message = counterBytes)`
4. `offset = hash[31] & 0x0f`
5. `codeInt = (((hash[offset] & 0x7f) << 24) | ((hash[offset + 1] & 0xff) << 16) | ((hash[offset + 2] & 0xff) << 8) | (hash[offset + 3] & 0xff)) % 1_000_000`
6. Render exactly six decimal digits, left-padded with zeroes.

This keeps the same time-bucket as rotating words and gives users who
prefer digits a familiar authenticator shape without making Safewords an
RFC 6238 account-token app.

## Codex redlines / decisions

### 1. Static override cannot be both deterministic and regeneratable

If the override is `HMAC(seed, "override")`, the user can reprint it from
the recovery phrase, but cannot invalidate old printed cards without
rotating the group seed. That is a feature, not a bug, for v1.3.

Decision: v1.3 static override is deterministic and reprintable, not
regeneratable. The UI should say "Reprint card" and "Rotate group seed to
replace this override." Do not offer "regenerate override" unless we add
stored override generation metadata plus a backup/migration story.

### 2. Challenge/answer should be group-table first

Counterparty selection sounds precise but is brittle in the current
product:

- Current groups do not have stable, cryptographic member identities.
- A recovered device may not know the same counterparty IDs.
- A stressed caller should not have to pick "which Alex" before verifying.

Decision: v1.3 challenge/answer uses a deterministic 100-row group table.
The app can choose a row for the user, or the user can choose one from a
printed card. Per-contact rotating pairs move to Pair mode.

### 3. Keep "Verify" as the product term, use "Match" as action copy

"Verify" matches the product promise and App Store naming. "Match" is
better button copy inside the challenge sheet.

Decision: nav/action title remains "Verify" where a nav item exists.
Inside the sheet use plain actions: "They said ORANGE" / "Does not
match".

### 4. Dynamic tabs are a secondary problem

Plain Mode as default means the primary UX should not depend on tab
math. Advanced view can keep stable tabs.

Decision: default home always shows contextual primitive actions for the
active group. Advanced/tabbed view shows Home, Groups, Settings always;
Verify appears only if any loaded group has a primitive that needs a
separate verify surface. If the active group does not support Verify,
show an explanatory empty state rather than hiding the tab while the user
switches groups.

### 5. Invite cards are seed-equivalent today

Until QR invites have expiry, recipient binding, or one-time semantics,
they are effectively recovery/invite secrets.

Decision: group invite card is high sensitivity and biometric-gated in
v1.3, or deferred.

## Codex answers to open questions

1. **Default primitive set for new groups.** Codex: rotating only. Offer
   override/challenge during setup as optional "printable backup" steps,
   but do not silently enable them.
2. **Static override regeneration.** Codex: no independent regeneration
   in v1.3. Reprint same deterministic override; rotate group seed to
   invalidate.
3. **Challenge/answer table size.** Codex: fixed 100 rows for v1.3.
   Deterministic from seed + tableVersion + rowIndex. No counterparty
   dimension yet.
4. **Override derivation.** Codex: deterministic from seed. Use a
   domain-separated label, not bare `"override"`.
5. **Verify tab nomenclature.** Codex: keep "Verify" as product/nav
   language; use "Match" in challenge confirmation copy.
6. **Migration safety.** Codex: yes. Missing `primitives` must decode to
   `rotatingWord.enabled = true`, interval from legacy group interval,
   all other primitives disabled. Add shared migration vectors.
7. **Tab visibility logic.** Codex: default Plain UI avoids tabs.
   Advanced view uses stable Home/Groups/Settings; Verify appears by union
   across loaded groups and handles unsupported active groups gracefully.

## Effort estimate (v3, updated for codex redlines)

- Plain-as-default migration (both platforms): 1-2 days each
- Group config schema + migration (with shared vectors): 0.5 day
- Static override primitive (logic + UI + reprint card): 1.5 days each
  *(reduced from 2 — no regeneration UI in v1.3)*
- Challenge/answer primitive (group-table + Match sheet + card): 2 days
  each *(reduced from 2-3 — no counterparty UX)*
- Native card rendering pipeline: 1.5 days each
- Recovery phrase printable card: 0.5 day each
- Numeric word format: 0.5 day each
- Shared schema + test vectors (`primitive-vectors.json`,
  `safety-card-copy.json`, `migration-vectors.json`): 1 day total
- Cross-platform test runs + parity check: 0.5 day

**Total**: roughly 8-10 days per platform, parallelizable.

## What this brief does NOT decide

- Visual design / branding for cards (use existing Ink theme on Android,
  default light on iOS for now; refine later)
- Exact onboarding copy changes (separate copy pass after structure
  lands)
- Whether to advertise primitives in App Store / Play Store listing
  (post-v1.3 marketing decision)

## Claude v3 deltas on top of codex redlines

All 9 codex redlines accepted. The following v3 clarifications resolve
remaining ambiguity so we can lock:

1. **Override word format**: same adj/noun/number phrase space, no
   dedicated wordlist. Resolved codex's "or" question.
2. **Numeric primitive pulled forward** to "Should" in v1.3. Cost: ~0.5
   day per platform. Benefit: occupies the Supercode search-term
   territory + offline-equivalent to TOTP authenticator apps for users
   who prefer digits.
3. **Challenge/answer printable**: full 100-row table is a "Full
   Protocol" card (folded letter); a 24-row excerpt fits the wallet
   card. Same derivation, two render templates. Wallet excerpt = first
   24 rows by table index, not random.
4. **Plain → Advanced reachability**: gear icon in Plain home top-right
   opens Settings (which lives in Advanced). User can toggle to
   Advanced view from there. Default is sticky — once a user picks
   Advanced they stay there until they toggle back.
5. **Verify-tab empty state copy** (Advanced view, group has no
   primitive needing Verify): "Verify isn't needed for *{group name}*
   right now — both phones show the same word. Add challenge/answer or
   a static override in Settings → Group → Primitives if you'd like a
   tap-to-confirm flow." Stable text, no scary error wording.
6. **Migration vectors example** for `migration-vectors.json`:

   ```json
   {
     "v1.2_to_v1.3": [
       {
         "name": "rotating-only group with default interval",
         "input": {
           "id": "00000000-0000-0000-0000-000000000001",
           "name": "Family",
           "seed": "0102...1f20",
           "intervalSeconds": 86400
         },
         "output": {
           "schemaVersion": 2,
           "id": "00000000-0000-0000-0000-000000000001",
           "name": "Family",
           "seed": "0102...1f20",
           "primitives": {
             "rotatingWord": {
               "enabled": true,
               "intervalSeconds": 86400,
               "wordFormat": "adjective_noun_number"
             },
             "challengeAnswer": { "enabled": false },
             "staticOverride": { "enabled": false }
           }
         }
       }
     ]
   }
   ```

   Both platforms must produce the v3 `output` shape from any v1.2
   `input` group.
